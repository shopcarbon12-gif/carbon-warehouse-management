Executive summary
  • Server APIs The same Coolify deployment described in the web parity report (68dbbd1 last finished) ships the Next.js routes and proxy.ts behavior the Flutter client depends on (Bearer + cookie, mobile paths, OTA upload, etc.).
  • Binary on phones ≠ git Each physical device runs whatever APK was installed (OTA or sideload). That version is only guaranteed to match main if you built from the same tree and distributed that artifact.
  • OTA row + file The server’s GET /api/mobile/status returns latestVersion and downloadUrl from Postgres (app_releases) and the file must exist under public/uploads/... on a persistent volume in Coolify. This session did not query your production DB or download the hosted APK.

1. Three-way model (source, server, handset)

Layer“Fully updated” meansHow to verify
A. Git / repo pubspec.yaml shows intended marketing + build number; Dart sources match features you expect. git pull, read pubspec.yaml, run flutter analyze locally.
B. wms.shopcarbon.com Deploy SHA includes mobile API routes + proxy rules; health OK. Coolify history; GET /api/health; optional authenticated API probes.
C. Device Installed package version matches the build you published; OTA URL returns a real APK bytes (not HTML 404/login). Android settings → app info; in-app Handheld settings “check OTA”; or adb shell dumpsys package com.shopcarbon.wms.

A gap at any layer looks like “production is wrong” even when the other two are correct — e.g. server updated but OTA file missing → WmsApiException(404) HTML; or server updated but phone still on old APK → missing UI flows.

4. Prior “APK vs git history” lesson (transcript thread)

In the APK build / SHA timeline chat, we established that an APK’s file timestamp does not prove inclusion of a git SHA unless the tree at flutter build time contained those commits. For production parity, the operational rule is: build → upload/set active → install on devices, and keep a record of the version label you published.

5. Checklist: confirm you are “fully updated”

  1. Coolify: Application on a commit ≥ 68dbbd1 (or latest finished after this report).
  2. Volume: Persistent storage mounted for uploads (OTA survives redeploy).
  3. OTA: Admin UI shows active release; label matches the binary you intend (e.g. 0.1.11+18).
  4. Handset: Settings → OTA status shows server label: and a non-empty download: URL when appropriate.
  5. Install: Device version string matches shipped APK; after install, exercise transfer slips + clean bin + lookup once.

6. Conclusion

Server-side, wms.shopcarbon.com at the last finished deploy (68dbbd1) carries the APIs and proxy behavior required for the current 0.1.11+18 source tree. Whether every handset is “fully updated” cannot be asserted from the repo alone: you must align OTA/database/file storage and installed binaries. Any remaining gap is most often artifact distribution or Postgres contents, not missing Next.js routes from the gap-closure batch.
