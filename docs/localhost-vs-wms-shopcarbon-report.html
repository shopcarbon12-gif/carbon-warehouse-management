CarbonWMS — Localhost vs wms.shopcarbon.com

Generated from repository state. Bins are not “synced” between environments by the application — they live in PostgreSQL. This report explains why localhost can show bins that production does not, how to verify deploy parity, and how to align data safely.

TL;DR — Why bins show on localhost but not on production
  • Expected Two separate databases. Local npm run dev uses DATABASE_URL from your machine (often Docker Postgres on localhost:5432). Production uses the Postgres instance linked in Coolify for the WMS app.
  • Bins are rows in the bins table, scoped by location_id → locations.tenant_id. The UI loads them via server components and GET /api/locations/bins using your session’s tenant (session.tid).
  • Running npm run seed (or bootstrap) on local only inserts Orlando-style bins into local Postgres — production never sees that unless you run equivalent SQL or use the WMS UI on production.
  • There is no built-in “push bins to cloud” job. The repo includes scripts/db-mirror-ssh-dump-restore.mjs for production → local mirror, not the reverse.

5. How to make production bins match local (or vice versa)

5.1 Recommended: use the WMS UI on production

Log in to https://wms.shopcarbon.com → Overview → Locations & bins and create the same bin codes you need. This avoids UUID / tenant mistakes.

5.2 Copy data with SQL (advanced)

Export locations / bins (and related rows if needed) from one DB and import into the other, rewriting tenant_id / location_id to match production’s UUIDs. Wrong IDs will create orphan rows or violate FKs. Prefer a one-off migration script reviewed by someone who knows your tenant layout.

5.3 Mirror production → laptop (see bins as on live)

Use scripts/db-mirror-ssh-dump-restore.mjs (documented in the script header) to replace local Postgres with a dump from the Coolify host. After that, localhost shows production’s bins, not the other way around.

6. Troubleshooting checklist

  • Different tenant Confirm you’re logged into the same logical warehouse/tenant on both hosts (memberships can differ per DB).
  • Stale deploy Redeploy WMS on Coolify after pushing main if you expect new API/UI behavior.
  • Production DB empty If production never ran seed/bootstrap, locations/bins tables may simply have fewer rows.
  • DATABASE_URL wrong in Coolify App may fall back to empty data or errors on overview pages — check /api/health/ready if you use DB readiness separately from liveness.
