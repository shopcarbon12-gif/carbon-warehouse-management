📱 Mobile App

Barcode Intake

A workflow for receiving new merchandise that does not yet have RFID tags. Staff scan the barcode on each item, the system records the receipt, and the item is queued for RFID tagging. This bridges the gap between barcode-only receiving and full RFID-tracked inventory.

Scan Interface

Barcode Scan Input
Scan / Text Input
A field that accepts barcode scans from the device's built-in barcode scanner or camera. When a barcode is scanned, the system looks up the corresponding item in the catalog. If found, the item is added to the intake session. If not found, an error or "unknown item" state is shown.
Suggestion: The scan field should give clear audio/visual feedback for each successful scan (a beep + green flash) and for failures (a different beep + red flash). In a busy receiving area with many items, staff need immediate confirmation without looking at the screen for every item.
Scanned Items List
List Display
Shows the items scanned in the current session with their names and quantities. Multiple scans of the same barcode increment the count for that item rather than creating duplicate entries.
Suggestion: Add a per-item remove button so staff can correct a mistaken scan without clearing the entire session. Show the running count clearly: "12 items scanned" at the top. Also, provide a way to add a manual quantity override for cases where a staff member is processing a large quantity of the same item (scanning 50 identical items one by one is slow).
Commit / Confirm Button
Primary Button
Finalizes the intake session and submits it to the server. After committing, the items are recorded as received and queued for RFID tagging. The session is then cleared.
Suggestion: The commit button should show a summary before finalizing: "Commit intake of 12 items? This will mark them as received and queue them for RFID tagging." Staff should confirm they haven't missed any items before committing. After commit, show a clear success screen with a count of items processed and an option to start a new intake session.

Overall Barcode Intake Suggestions

1. No draft/pause feature: If a staff member needs to step away mid-intake (break, question, distraction), the session should be saveable as a draft. Losing progress on a large receiving batch is frustrating.

2. No duplicate detection within session: If the same barcode is scanned twice on different items (e.g., two of the same SKU that need separate serial numbers), the system should handle this gracefully — counting them as two units, not flagging as an error.

3. Integration with receiving workflow: Ideally, barcode intake should link to a purchase order or delivery receipt so the received items can be matched against what was expected. Currently it appears to be a freeform intake without expected-quantity validation.
