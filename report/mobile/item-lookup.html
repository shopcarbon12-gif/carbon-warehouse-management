📱 Mobile App

Item Lookup

Allows floor staff to search for an item by scanning its RFID tag or barcode, or by typing part of its description or SKU. Returns the item's current status, location, and details. Primarily used when a staff member needs to quickly find information about a specific item on the floor.

Search Input

Search / EPC Input
Text Input
A text field where the user can type a search query. The search matches against EPCs (the RFID tag codes), barcodes, SKU numbers, and item descriptions. Scanning an RFID tag on a physical item automatically fills this field with the EPC code. The search is triggered after the user types or a tag is scanned.
Suggestion: The search field should have a clear placeholder explaining what it accepts: "Scan a tag, or type an EPC, barcode, or item name." Also, after a scan populates the field and results appear, the field should stay visible (not auto-dismiss the keyboard) so the user can easily scan another item.

Search Results

Item Result Card
Result Display
When a match is found, shows the item's details: product name, SKU, EPC code, current status (one of the Clean 10 statuses), the bin it is assigned to, and any other relevant metadata. For a direct EPC match (exact tag scanned), the result is shown immediately without a results list. For text searches, a list of matching items may appear.
Suggestion: The item result card should make the most important information visually prominent — the item's current status should be the biggest/most visible piece of information, using color coding (green for in-stock, red for damaged/stolen, etc.). Staff most often use Item Lookup to answer "is this item supposed to be here?" — the status answer should be immediately obvious without reading.
EPC Profile Matching
Data Processing
When an RFID tag is scanned, the app checks its EPC code against the configured encoding profiles to decode the product ID and serial number from the tag's bit pattern. If the code matches a known profile, the decoded item is shown. If no profile matches, an unknown tag message appears.
Suggestion: When a tag cannot be matched to any known profile, the error message should be specific: "This tag was not encoded by this system" vs "This tag exists in the system but could not be decoded." These are different problems with different solutions. The first means the tag came from another system; the second means there may be a profile configuration issue.
No Results State
State
When no matching items are found, a message appears indicating that nothing was found for the search query.
Suggestion: The "no results" state should suggest next steps: "No item found for this EPC. If this is a new tag, it may need to be commissioned first. Or try searching by item name or SKU." This guides staff rather than leaving them with a dead end.

Overall Item Lookup Suggestions

1. No status change action from here: When a staff member looks up an item and sees it has the wrong status, they currently have to navigate to the Change Status screen separately. A shortcut action on the result card ("Change this item's status") would streamline the workflow.

2. No history of recent lookups: Staff frequently look up the same items repeatedly during a shift. A "Recent lookups" list on the search screen (showing the last 5–10 items) would save time and allow quick re-access.

3. No location context in result: The result shows what bin an item is assigned to, but not whether the item was found in the right place. For cycle-count-style use, showing "Expected bin: A3 — Currently assigned to: A3 ✓" or "Expected bin: A3 — Currently assigned to: B7 ⚠" would be very useful.
