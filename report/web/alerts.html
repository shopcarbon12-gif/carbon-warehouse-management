Page Header Links

Link to Operations → Exceptions & Audit Log
Navigation Links
Links near the top: one to the Operations floor workspace, one to the Audit Log report. These cross-link the two related exception views and the audit trail.
Cross-linking the admin view with the floor workspace and audit log is the right pattern — admins reviewing exceptions can jump between these contexts without navigating through the sidebar.
The audit log link points to "/reports/audit" which may not exist as a direct route — it might silently redirect or 404. A broken link on an admin page erodes trust in the system.
Verify the audit log link is functional. Add a brief description next to each link explaining which audience it serves: "Floor workspace (for real-time handling) · Audit log (for full history and compliance)."

Exceptions Table

Exceptions Table
Data Table
Loads all exceptions from the database on page load. Shows four columns: Type, Detail, State (new/assigned/resolved/ignored), and Actions. Data auto-loads when the page opens.
The table has no filtering or sorting. An admin reviewing exceptions needs to filter by type, date range, and state. Loading all exceptions into one unfiltered list becomes unusable as the system matures.
The page loads data once on open and has no auto-refresh. New exceptions that arrive after the page loads are invisible until the admin manually reloads the page.
Add filter controls: by exception type, by state (new/resolved/ignored), and by date range. Add a manual refresh button and a "last updated" timestamp. Consider a 30-second auto-refresh interval for this admin view.
Resolve Button (per row)
Action Button
Only shown for "new" or "assigned" exceptions. Clicking immediately changes the state to "resolved" and refreshes the table. No confirmation, no note required.
Resolving an admin-level exception with one click and no note creates an empty audit trail. The record says "resolved" with no explanation of what was done or by whom.
Require a brief note field before resolving: "Resolution note (required)." This is especially critical for admin-level actions where accountability matters. The note should appear in the audit log.
Ignore Button (per row)
Action Button
Only shown for "new" or "assigned" exceptions. Clicking changes the state to "ignored" and refreshes. No confirmation, no reason required.
"Ignored" exceptions look the same as "Resolved" in the table — there is no visual distinction. Ignoring an exception (decided it's not real) vs resolving it (took action) are fundamentally different outcomes.
Color-code the state pill: "Resolved" in green, "Ignored" in grey/amber, "New" in red. Require a reason for ignoring as well — "Why is this being ignored?" prevents legitimate issues from being silently dismissed.
Empty State
Empty State
When there are no exceptions, the table shows: "No exceptions — run compare or seed data." This message contains developer language ("seed data").
"Seed data" is developer jargon with no meaning to an admin. This message must not appear in a production environment — it exposes how the system was developed rather than guiding the user.
Replace with: "No open exceptions. Everything looks good." Remove all developer instructions from the production empty state. Add a note for when exceptions do appear: "Exceptions are generated automatically by system scans and comparisons."

Overall Assessment

The admin exceptions view has the right data but is missing the controls admins actually need. No filtering, no required notes on resolution, and developer-facing empty state text all need to be addressed before this page is genuinely useful for admin oversight.
Key improvements:
1. Add filtering by type, state, and date range.
2. Require a note/reason when resolving or ignoring — build the audit trail properly.
3. Visually distinguish "Resolved" vs "Ignored" states with color coding.
4. Remove all developer language ("seed data") from empty states.
