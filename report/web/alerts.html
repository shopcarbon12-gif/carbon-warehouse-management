Alerts & Exceptions (Admin View)

An admin-level table listing all system exceptions. Unlike the Operations → Exceptions floor workspace (which is designed for floor supervisors), this page is for managers and admins to get a broader view. Each change made here creates an audit record.

ℹ️ This page is separate from Operations → Exceptions. Both show exceptions but with different audiences in mind. This page is the admin-level view; the Operations page is for floor staff.

Page Header Links

Link to Operations → Exceptions
Link
A teal link labeled "Operations → Exceptions" near the top of the page. Takes you to the floor workspace for handling exceptions in real time.
Suggestion: Good that the two pages are cross-linked. However, users may be confused about which page to use. A brief description like "For real-time floor work, use the Operations workspace. This page is for admin review and bulk actions." would clarify the difference.
Link to Audit Log
Link
A link to "Audit log" which should take you to the audit/activity report. Note: the link points to "/reports/audit" which may not exist as a direct route — it might redirect to the activity history report.
Suggestion: Verify this link actually works and points to the correct page. A broken link on an admin page erodes trust in the system.

Exceptions Table

Exceptions Table
Data Table
Loads all exceptions from the database on page load. Shows four columns: Type (exception category), Detail (description of what happened), State (current status: new, assigned, resolved, or ignored), and Actions. Data auto-loads when the page opens — no manual refresh button is visible.
Suggestion: The table has no filtering or sorting. An admin reviewing exceptions needs to be able to filter by type, date range, and state. Also, the table auto-refreshes only on page load — adding an auto-refresh interval (like the Operations page) would keep the admin view current without needing to reload the page.
Resolve Button (per row)
Button
Only shown for exceptions in "new" or "assigned" state. Clicking immediately changes the exception's state to "resolved" and refreshes the table. No confirmation dialog, no reason/note required.
Suggestion: Resolving an exception without leaving a note creates an incomplete audit trail. The system should require at minimum a brief reason: "Why is this resolved?" with a short text field. This is especially important for admin-level actions.
Ignore Button (per row)
Button
Only shown for exceptions in "new" or "assigned" state. Clicking changes the state to "ignored" and refreshes the table. Again — no confirmation, no reason required.
Suggestion: "Ignore" should also require a brief note. Additionally, ignoring an exception should be distinguishable from resolving it — "Ignored" exceptions should be visually different from "Resolved" ones in the table (e.g., different color tag).
Empty State
Empty State
When there are no exceptions, the table shows: "No exceptions — run compare or seed data." This again exposes developer language ("seed data") in a user-facing context.
Suggestion: Replace with: "No open exceptions. Everything looks good." Remove the developer instruction entirely from production.

Loading State

Loading Indicator
Loading State
While data loads, the page shows "Loading…" in the table area. There is no error handling shown if the API call fails — the table simply stays empty if the API doesn't respond.
Suggestion: Add error state handling: "Could not load exceptions. Please refresh the page or contact support if the problem persists."
