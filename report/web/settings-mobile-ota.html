Mobile OTA Releases

Manages over-the-air (OTA) updates for the CarbonWMS handheld Android app. Admins upload new APK files here, which are then pushed to handhelds. The "active" release is what handhelds download when they check for updates.

Upload New Release

Version Label Input
Text Input
A text field for the version number of the APK being uploaded. If you drag and drop an APK with a version in its filename (like "CarbonWMS v1.1.1+28.apk"), the system automatically detects and fills in the version label. Format expected: "v1.1.1+28".
Suggestion: The auto-detection from filename is a great UX touch. Extend this to also auto-detect the version from inside the APK manifest file when uploaded, as a more reliable source. Also validate that the version is newer than the current active release to prevent accidentally downgrading.
APK File Drop Zone / Browse Button
File Input
A drag-and-drop zone for the APK file. Can also be clicked to open a file browser. Only accepts ".apk" files. When a valid file is dropped, shows the filename and file size. Invalid file types show an error message.
Suggestion: Show a file size warning if the APK is unusually large (over 150MB), as this might indicate an error. Also show a checksum (MD5/SHA256) of the uploaded file so admins can verify it matches the build they intended to upload.
Upload Button
Primary Button
Uploads the selected APK file along with the version label. Shows "Uploading [filename]…" during the upload. On success, the releases list refreshes to show the new entry. On failure, an error message appears.
Suggestion: Large APK files can take time to upload. Show a progress bar with percentage (not just "Uploading…") so the admin knows how much longer to wait. Also, if the upload is interrupted, a retry option should appear.

Releases List

Releases Table
Data Table
Shows all uploaded APK releases. Columns: version label, upload date, whether it is the active release (the one handhelds will download). Each row has a "Set active" button to make that version the active release.
Suggestion: Clearly distinguish the currently active release visually — a green "ACTIVE" badge or highlighted row. Also show download count per release so admins can confirm handhelds are actually downloading the update.
Set Active Button (per release)
Button
Sets the selected release as the one that will be served to handhelds via the OTA update API. Previous active release becomes inactive.
Suggestion: Setting a new active release should show a confirmation: "Make v1.1.1+28 the active release? All handhelds will be prompted to update the next time they check." Also consider a scheduled activation — "Activate at 11 PM tonight" for off-hours rollouts.

Overall Mobile OTA Suggestions

1. Rollback safety: Clearly mark the previous active version so it can be quickly restored if the new version has issues on the floor.

2. Update adoption tracking: Show how many of the authorized devices have already updated to the latest version (e.g., "7 of 12 devices updated").

3. Release notes: Add a "Release notes" or "What's new" field when uploading so managers know what changed in each version without needing to ask developers.
