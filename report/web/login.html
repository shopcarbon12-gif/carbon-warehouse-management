🌐 Web App

Login Page

The first thing anyone sees when visiting the system. It is a centered card with an email and password field. If you are already logged in, you are automatically sent to the Dashboard (or wherever you were trying to go).

Page Layout

Developer Notice Banner (top of page)
Informational Banner
A "Dev Local DB" banner can appear above the login card. This is only shown in development mode (when the system is connected to a local database instead of the live production one). It warns developers they are not looking at real data.
Suggestion: This banner is for developers only. Make sure it is completely hidden in production so it does not confuse or concern warehouse staff. A brightly colored warning banner on a login screen can alarm users unnecessarily.
Login Card
Container
A white/dark bordered box centered on the screen. Contains the "Sign in" title, a small note about what database the system is using, and the login form. Maximum width is 384px (about the width of a phone screen), so it is easy to read.
Suggestion: The subtitle under "Sign in" currently says "Sign in with a user that exists in the database behind DATABASE_URL (local seed or production)." — This is a technical message intended for developers, not for warehouse staff. It should be replaced with something friendly like "Enter your work email and password to continue."

Form Fields & Buttons

Email Field
Input Field
A standard email input. Has placeholder text showing an example email. Marked as required, so the browser will show a warning if you try to submit without filling it in. Uses autocomplete to suggest saved emails from the browser.
Suggestion: The placeholder shows "e.g. admin@example.com (local seed)" — again, this is developer language that should be replaced with something like "Your work email" for production. The field label says just "Email" in very small light grey text, which might be hard to read in bright warehouse lighting.
Password Field
Input Field
A standard password field that hides the typed characters. No "show password" toggle button is visible on the web version. It uses browser autocomplete to suggest saved passwords. Required field.
Suggestion: There is no "show/hide password" eye button on the web login. This is a very standard UX feature — users often make typos in the password field and cannot see what they typed. Adding a toggle to reveal the password would reduce login failures. Also, there is no "Forgot password" link — if a user forgets their password they have no self-service option.
Sign In Button
Primary Button
A solid teal/green button that spans the full width of the form. While signing in, it changes text to "Signing in…" and becomes disabled (greyed out) to prevent double-clicks. After a successful login, the user is redirected to the Dashboard or to the page they originally tried to access.
Suggestion: The loading state shows "Signing in…" which is clear. However, if the server is slow, there is no timeout message — the button just stays in the loading state forever. Adding a timeout (e.g., 30 seconds) with a message like "This is taking longer than expected…" would reduce user frustration.
Error Message
Error Display
If login fails (wrong password, server error, etc.), a small red text message appears below the password field explaining what went wrong (e.g., "Login failed" or "Network error"). The message disappears when you try again.
Suggestion: The error messages are short but not very helpful. "Login failed" does not tell the user if they typed the wrong password or if their account is locked. Consider distinguishing between "Incorrect email or password" and "Your account has been disabled." Also, after 5 failed attempts, consider adding a brief cooldown period to prevent password guessing.

Automatic Redirect Behavior

Already Logged In Redirect
Page Behavior
If you visit the login page while already signed in, the system immediately redirects you to the Dashboard without showing the login form at all. If you tried to access a specific page before being redirected to login, the system remembers where you were going and takes you there after you sign in (using a "next" URL parameter).
Suggestion: This is good and expected behavior. However, if the session has expired and the user is in the middle of doing something (like filling in a form), they lose all their unsaved work when redirected to login. A more graceful approach would be to show a "Your session has expired, please sign in again" modal instead of a hard redirect, then return them to exactly where they were.

Overall Login Page Suggestions

1. Clean up developer text: Remove all developer-facing language (database hints, example emails) for the production version.

2. Add show/hide password toggle: Standard feature, reduces login frustration.

3. Add "Forgot password" link: Even if it just shows an admin contact email, this is better than nothing.

4. Add company branding: The login page shows just "Sign in" with no logo or company name. Adding the Carbon logo and company name would make it feel more professional and help users confirm they are on the right system.

5. Better error specificity: Help users understand what went wrong without giving away security information.
