Cycle Counts

A physical inventory count tool. Select a location and bin, then scan RFID tags (or simulate scanning). The system compares what was scanned against what should be there and classifies each tag as matched, missing, misplaced, or unrecognized. Then you commit the count to record the results.

Location & Bin Selection

Location Dropdown
Dropdown
Populated from the system's location registry. Selecting a location updates the bin dropdown to show only bins belonging to that location. Required before starting a count.
Suggestion: The active session location should be pre-selected by default to save time. Most users will be counting the location they are already working in.
Bin Dropdown
Dropdown
Populated after selecting a location. Shows all bins at that location along with their current in-stock count (so you can see "Bin A4 — 23 items" before starting). Selecting a bin loads the expected items for that bin.
Suggestion: Showing the in-stock count in the dropdown is helpful — it sets the expectation for how many items should be found. Consider also showing a "last counted" date per bin to help prioritize which bins need counting.

Scanning Controls

Start / Stop Scan Button
Toggle Button
A button that toggles RFID scanning on/off. When scanning is active, the system listens for incoming RFID reads via a live stream. All detected EPCs are added to the scanned list in real time. The button changes label and style when active.
Suggestion: There should be a clear visual indicator that scanning is currently active — not just a button label change. A pulsing green "SCANNING LIVE" banner at the top of the screen would be much more obvious, especially when the user is looking at the results list rather than the button.
Shuffle / Simulate Button
Button
A "shuffle" button (icon) that simulates a scan by randomly marking some expected EPCs as misplaced or unrecognized. This is a demo/testing feature — it lets you see the classification system in action without real hardware.
Suggestion: This simulation button should not be visible in a production environment — it could confuse warehouse staff who might click it accidentally and think they have a real scan result. Either hide it in production mode or label it very clearly as "TEST MODE - Simulate scan."
Clear / Reset (implied)
Button
Presumably there is a way to clear the scanned list and start over. This would be needed when a count was started accidentally or the wrong bin was selected.
Suggestion: Make sure there is a clearly labeled "Start over" or "Clear scanned items" button with a confirmation dialog, since clearing a partially completed count cannot be undone.

Classification Results

Matched Items
Result Category
Items that were in the expected list AND were scanned. These are correct — the item is where it should be.
Suggestion: Show the matched count prominently with a green indicator. A count like "23 of 30 expected found" gives instant status.
Missing Items
Result Category
Items that were expected in this bin but were NOT scanned. Either they are genuinely missing, or the RFID reader missed them (possible with tags in difficult-to-read positions).
Suggestion: Highlight missing items in amber/red. For each missing item, show its last known location and last scan date so staff can investigate. Also note that RFID can miss reads, so consider adding a "Re-scan missed items" button before treating them as missing.
Misplaced Items
Result Category
Items that were scanned but belong to a DIFFERENT bin (they are in the wrong place). The system can detect this because it knows which EPCs should be in which bins.
Suggestion: For each misplaced item, show where it should be (the correct bin) alongside where it was found. This makes it easy for staff to know where to move it.
Unrecognized Items
Result Category
RFID tags that were scanned but do not exist in the system at all. Could be tags from a different company, corrupted tags, or items that were never properly commissioned.
Suggestion: Unrecognized tags should automatically generate an "Unknown Asset" exception that appears on the Alerts page, so they don't just get counted and forgotten.

Commit Count Modal

Commit Button → Modal
Button → Modal
Opens a confirmation modal summarizing the variance before it is recorded. Shows totals for matched, missing, misplaced, and unrecognized items. Confirming the commit writes the count results to the database as an official inventory count.
Suggestion: The commit modal should clearly state the consequences: "Committing this count will update inventory records. Missing items will be flagged for investigation. Misplaced items will need to be physically relocated." This helps staff understand the weight of confirming the operation.
