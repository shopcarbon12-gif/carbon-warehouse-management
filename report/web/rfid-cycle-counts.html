Cycle Counts

A physical inventory count tool. Select a location and bin, then scan RFID tags (or simulate scanning). The system compares what was scanned against what should be there and classifies each tag as matched, missing, misplaced, or unrecognized. Then commit the count to record the results.

Recommendations

Location & Bin Selection

Location Dropdown
Dropdown
Populated from the system's location registry. Selecting a location updates the bin dropdown to show only bins belonging to that location. Required before starting a count.
The active session location is not pre-selected by default — every count starts by making the user manually pick the location they are already working in.
Pre-select the user's active location automatically. Save the last-used location to session/localStorage so returning to this page after a break doesn't require re-selecting.
Bin Dropdown
Dropdown
Populated after selecting a location. Shows all bins at that location along with their current in-stock count (e.g., "Bin A4 — 23 items") before starting. Selecting a bin loads the expected items for that bin.
Showing the in-stock count in the dropdown sets the expectation for how many items should be found before the count even starts — this is a useful planning cue for the operator.
Add a "last counted" date per bin in the dropdown: "Bin A4 — 23 items — Last counted: 14 days ago." This helps prioritize which bins need counting most urgently.

Scanning Controls

Start / Stop Scan Button
Toggle Button
A button that toggles RFID scanning on/off. When active, the system listens for incoming RFID reads via a live stream. All detected EPCs are added to the scanned list in real time. The button changes label and style when active.
The only indication that scanning is active is the button label change. When the operator is looking at the results list rather than the button, there is no obvious "I am currently scanning" signal.
Add a pulsing green "SCANNING LIVE" banner at the top of the screen while scanning is active — visible regardless of where on the page the user is looking.
Shuffle / Simulate Button
Demo Button
A "shuffle" button that simulates a scan by randomly marking some expected EPCs as misplaced or unrecognized. Used for demo/testing — lets you see the classification system without real hardware.
This testing button is visible in what appears to be the production environment. Warehouse staff could click it accidentally and produce a fake scan result they believe is real.
Hide this button entirely in production mode, or add a very obvious label: "TEST ONLY — Simulate scan" with a confirmation warning: "This will create fake scan results. For testing only."

Classification Results

Matched Items
Result Category
Items that were in the expected list AND were scanned. These are correct — the item is where it should be.
The 4-category classification (matched / missing / misplaced / unrecognized) is exactly the right model for RFID cycle counting. Each category has a distinct operational meaning and response.
Show the matched count as a progress indicator: "23 of 30 expected found" so the operator knows at a glance whether to keep scanning or if the count is likely complete.
Missing Items
Result Category
Items expected in this bin but NOT scanned. Either genuinely missing, or the RFID reader missed them (possible with tags in difficult positions).
Missing items and unrecognized items look the same visually — both appear as plain list entries. Staff cannot quickly distinguish "item might be behind a shelf" from "this tag is from another company."
For each missing item, show its last known location and last scan date. Add a "Re-scan missed items" quick action before treating them as confirmed missing, since RFID can miss reads.
Misplaced Items
Result Category
Items that were scanned but belong to a DIFFERENT bin (they are in the wrong place). The system can detect this because it knows which EPCs should be in which bins.
The misplaced items list shows the item and where it was found, but doesn't immediately show where it should be — requiring the operator to look it up separately.
For each misplaced item, show two pieces of information side by side: "Found here: Bin A4 → Should be in: Bin B2." This makes the relocation task immediate and actionable.
Unrecognized Items
Result Category
RFID tags that were scanned but do not exist in the system at all. Could be tags from a different company, corrupted tags, or items never properly commissioned.
Unrecognized tags are simply listed in the count results with no automatic follow-up. They can be committed and forgotten, with no alert generated.
Unrecognized tags should automatically generate an exception on the Alerts page when committed, so they are investigated rather than silently counted and discarded.

Commit Count Modal

Commit Button → Modal
Button → Modal
Opens a confirmation modal summarizing the variance before it is recorded — totals for matched, missing, misplaced, and unrecognized. Confirming the commit writes the count results to the database as an official inventory count.
The commit modal shows a summary before recording — operators can see matched/missing/misplaced totals before making a permanent decision. This is the right pattern for a high-impact action.
The commit modal should clearly state the real-world consequences: "Committing this count will flag 3 items as missing for investigation and will update bin records for 5 misplaced items. This cannot be undone."

Overall Assessment

Cycle Counts is one of the most complete RFID tools in the app. The 4-category classification, the in-stock count preview in the bin dropdown, and the commit confirmation modal are all solid design decisions.
Key improvements:
1. Add a visible "SCANNING LIVE" banner — the button label alone is not enough.
2. Remove or clearly hide the simulation button from production.
3. Show "where it should be" directly next to misplaced items.
4. Auto-generate exceptions for unrecognized tags on commit.
