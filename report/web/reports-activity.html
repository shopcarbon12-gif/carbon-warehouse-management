🌐 Web App

Activity History Report

Shows a scrollable log of the last 200 system events in reverse chronological order. Every action taken in the system — tag scans, status changes, logins, imports, exception resolutions — generates an entry here. Auto-refreshes when the page regains focus.

The Activity Table

Activity Table
Data Table
Two columns: "When" (exact timestamp in the user's local time zone, formatted to the second) and "Event" (description of what happened). Events are formatted as concatenated strings: "action · entity — detail." The table shows up to 200 rows. Newest events appear first.
Suggestion: The event format "update · rfid_asset — E280..." is technical and machine-readable. For a report that managers review, events should be written in plain language: "RFID tag E280... moved to Bin A4" or "User admin@carbon.com resolved exception #123." Consider translating event types to human-readable descriptions.
When Column (Timestamp)
Table Column
Shows exact date and time of each event, formatted using the browser's locale. Uses a monospace font for easy reading of timestamps.
Suggestion: Add relative time ("2 minutes ago," "Yesterday at 3:45 PM") on hover or as a secondary label. Also consider showing which user performed the action — currently there is no "Who" column.
Loading State
Loading
Shows "Loading…" while fetching data. No error state is visually distinct — if the API call fails, the table may just show "No audit rows."
Suggestion: Distinguish between "No audit rows (nothing has happened yet)" and "Failed to load data (network error)." These require different user actions.
Empty State
Empty State
Shows "No audit rows." when the table is empty.
Suggestion: Clarify: "No activity recorded yet. Events will appear here as actions are taken in the system."

Overall Activity History Suggestions

1. No filtering: 200 rows of mixed events are hard to review. Add filter by: date range, event type (scan / status change / login / import), and user. This turns the log from a raw dump into a useful audit tool.

2. No export: Auditors often need this data in a spreadsheet. Add a "Download CSV" button.

3. Hard limit of 200: For compliance purposes, all events should be accessible. Add pagination or a date-range filter so older events can be retrieved.

4. No "Who" column: Every audit event should show which user performed the action. This is essential for accountability.
