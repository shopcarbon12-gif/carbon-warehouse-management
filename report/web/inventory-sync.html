Lightspeed Sync

The page for synchronizing the product catalog from Lightspeed POS into CarbonWMS. Has two tabs: "Sync Engine" (manual sync + history) and "Live Compare" (real-time POS vs RFID comparison).

Sync Engine Tab

Last Success Tile
Status Display
Shows the exact date and time when the last successful sync completed (pulled real data from Lightspeed). If a sync has never been done successfully, it shows "—". Auto-refreshes every 20 seconds.
Suggestion: Consider adding a "time ago" label: "Last synced 2 hours ago" is more immediately meaningful than an exact timestamp. Also, if the last sync was more than 24 hours ago, the tile should turn amber to signal the data might be stale.
Synchronized SKUs Count Tile
Status Display
Shows the total number of SKU rows currently in the catalog database. The technical description mentions "custom_skus rows in the matrix catalog" — this is internal language. Auto-refreshes every 20 seconds.
Suggestion: The technical note "Total custom_skus rows in the matrix catalog" should be hidden from regular users. A simpler label like "Products in catalog" is sufficient. Also, this number alone is not enough context — was the last sync partial or complete? Did it add new items or remove discontinued ones?
Trigger Manual Sync Button
Primary Button
A full-width teal button with a refresh icon. Clicking it immediately starts a sync with Lightspeed. The button shows a spinning icon and says "Syncing…" while the process runs. After completing, a summary message appears showing: how the sync finished, how many SKU rows were updated, where the data came from (e.g., "R-Series", "simulated"), and any warnings.
Suggestion: The sync source label "simulated" is a red flag that should be very visible — if the system is using fake data because real credentials are not configured, the page should display a prominent warning banner rather than a small text note in the result message. Also, the sync has no visible progress bar — for large catalogs, users don't know if it is running for 5 seconds or 5 minutes.
Sync History Log
Table / Timeline
Below the sync panel, a history of past sync runs is shown. Includes timestamp, outcome (success/failure), source, and number of records updated. This section loads separately from the main sync panel.
Suggestion: If sync jobs are running automatically in the background (via a scheduled worker), those should show up in the history too, not just manually triggered ones. Also, failed syncs should show the error reason so users can diagnose problems.

Live Compare Tab

Compare Engine / POS vs RFID
Comparison Tool
Compares POS (Lightspeed) on-hand quantities against RFID in-stock EPC counts at the active location. Shows per-SKU discrepancies. The description notes this uses "mock" POS data in some configurations.
Suggestion: If the data being compared is "mock" (simulated/fake), this should be displayed prominently with a warning banner. Users might take corrective actions based on fake discrepancy data, which would cause real problems. Also, the comparison should clearly show the date/time of the POS data used for the comparison.
Background Jobs / Sync Panel
Worker Jobs
Shows scheduled background sync jobs queued in the database. The description says to run "npm run worker" to process them — a developer instruction that should not appear on a user-facing page.
Suggestion: Remove all developer instructions from the production UI. Replace with a simple status indicator showing whether the background worker is running and when it last processed jobs. If the worker is not running, show an alert.
