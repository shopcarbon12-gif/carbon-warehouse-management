Operations — Exceptions (Floor Workspace)

The main workspace for floor supervisors to handle RFID alarms and exceptions — dock alarms (items leaving without authorization), unexpected tag reads, or system anomalies. Refreshes automatically every 10 seconds.

Filter & Controls

OPEN / RESOLVED / ALL Filter Buttons
Filter Toggle
Three buttons to filter the exceptions list. Default is "OPEN" — unresolved items. "RESOLVED" shows dealt-with items. "ALL" shows everything.
Defaulting to "OPEN" is the right call — a floor supervisor opening this page wants to see what needs action immediately, not a full history of resolved issues.
The filter buttons show no counts. A supervisor must switch to each view to understand the scope: how many open, how many resolved.
Add live counts to each filter tab: "OPEN (4) · RESOLVED (23) · ALL (27)." This gives instant triage information at a glance without switching views.
Simulate Dock Alarm Button
Danger Button
A button styled as a warning action that creates a simulated dock alarm exception in the database. Used for testing or demonstrations. Shows "Working…" while running and then shows a toast with the alarm ID.
This testing button exists in what appears to be a production UI. Accidentally clicking it adds a fake alarm to the real system, forcing someone to resolve a non-existent exception.
Hide this button in production mode entirely. If it must be kept, add a very obvious "TESTING ONLY" label and require a confirmation: "This will create a fake exception in the live system. Continue?"
Refresh Button
Icon Button
Manually refreshes the exceptions list. The list also auto-refreshes every 10 seconds, so this is for users who want to check immediately.
The 10-second auto-refresh is the right interval for a floor operations page — it keeps the data fresh without requiring manual action or overwhelming the server.
Show a "Last updated X seconds ago" indicator near the refresh button so supervisors know the auto-refresh is working without clicking manually.

Exceptions Table

Exceptions List
Data Table
Shows exception rows with: exception type, timestamp, current state (open/resolved), and who handled it. Clicking a row opens the resolution modal. Sorted most recent first. Rows are color-coded by severity.
Exception type names are technical internal codes like "DOCK_EXIT_UNAUTHORIZED." Floor staff reading this during a busy shift cannot immediately understand what happened.
Translate exception types into plain language in the table: "Item left building without authorization" instead of "DOCK_EXIT_UNAUTHORIZED." Show the item name and last known location directly in the row so the supervisor can act without opening the detail modal.
Row Click → Resolution Modal
Modal
Clicking a row opens a detailed modal. Shows full exception details and allows setting a resolution — adding a note and marking as resolved. Once resolved, it disappears from the OPEN view.
Resolution can be submitted with no note or explanation. This creates an audit trail that says "resolved" with no record of what happened or why.
Require a brief reason before allowing resolution — a short free-text field: "What happened? (required)" The modal should also show related information: which specific item triggered the alarm, its current status, and its last known scan location.

Overall Assessment

The Operations Exceptions page is a solid floor workspace. The auto-refresh, severity color-coding, and open/resolved filtering are all the right building blocks. The main gap is that the content itself (exception names, empty notes on resolution) isn't ready for non-technical floor staff.
Key improvements:
1. Add counts to filter tabs — how many open vs resolved.
2. Translate exception type codes to plain language in the table.
3. Require a note/reason when resolving an exception.
4. Remove or clearly gate the "Simulate dock alarm" button from production.
