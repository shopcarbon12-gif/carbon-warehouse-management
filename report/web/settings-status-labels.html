Status Labels (Clean 10)

Manages the 10 fixed item statuses that exist in CarbonWMS — known as the "Clean 10." Each status has a set of rules that are permanently fixed (such as whether the status counts as sellable, whether scanners respond to it, and whether staff can see it). Admins can only change the display name and a legacy system ID number for each status.

Status Table

Status Rules Table
Data Table
Displays all 10 statuses in a table. Columns: Status name, System ID (a number used by integrations), Web sellable (Yes/No — whether items in this status count toward available quantity), Scanner (Yes or "Ignore" — whether RFID readers respond to tags in this status), UI visible (Yes/No — whether this status appears in search and reports), Icons (a set of three emoji indicators), and an Edit button. The icons shown in a small badge panel are: 👑 (Super Admin lock — staff cannot change items out of this status without a super admin), 👻 (Ghost — scanner ignores reads completely, no beep, no count), and 🤖 (System only — hidden from staff's status change options). Empty icon slots show a dash.
Suggestion: The emoji icons (👑 👻 🤖) are clever but not self-explanatory to new admins. Add a legend below or above the table explaining what each icon means. The tooltips exist on hover, but many users never discover hover text. A simple legend card ("👑 = Super Admin required to change · 👻 = Scanner ignores these items · 🤖 = Hidden from staff menus") would make this page much clearer.
View / Edit Label Button
Button
Opens a detailed popup for that status. The popup shows: the status's internal name, a description of what the status means, a full "Hard-wired rule summary" panel (read-only) listing all its flags and what they mean in plain language, an editable "Display label" field (an optional custom name that appears in the UI instead of the internal name), and an editable "Legacy system ID" field (a number used by older connected systems). Saving updates only these two fields. The behavior rules themselves cannot be changed.
Suggestion: The popup layout is well-structured. One improvement: the "Hard-wired rule summary" panel uses phrases like "Web sellable: Yes — quantity sync follows this flag." This is clear, but could be even simpler. Also, the "Display label" field could use a placeholder example, e.g., "e.g. In Stock (or leave empty to use the default name)." Finally, show a preview of how the label will look in the inventory table while typing.

Status Descriptions Reference

The 10 fixed statuses and what they mean to warehouse staff (for reference in any documentation or training):

in-stock: Normal item present in the warehouse, counts as sellable.
return: Item returned by a customer, may need inspection.
damaged: Item is damaged; should not be sold.
sold: Item has been sold at the POS; scanner ignores it (ghost).
stolen: Item flagged as stolen; Super Admin lock applies.
tag_killed: RFID tag has been intentionally deactivated (e.g., at checkout); scanner ignores it.
UNKNOWN: Status is not yet determined; system-only.
pending_visibility: Item is in the system but not yet confirmed in stock.
in-transit: Item is moving between locations.
pending_transaction: Item is part of a pending sale or transfer, not yet finalized.

Suggestion: These descriptions (or similar ones) should appear on the status labels settings page itself so admins understand what each status represents. New admins often ask "what is UNKNOWN?" or "why is tag_killed separate from sold?" — a brief description on this page would answer those questions.

Overall Status Labels Suggestions

1. Read-only but not explained: Most of the important fields on this page cannot be edited, but this is not explained until the user clicks Edit. Consider adding a brief note at the top: "The behavior rules for each status are fixed by the system. You can only change the display name shown to staff and the legacy ID for integrations."

2. No search or filter: With only 10 statuses, filtering is not urgent, but as this page grows (if statuses are ever expanded), basic sorting by name or flag would be useful.

3. Icon legend missing from the page: As noted above, the emoji icons need a visible legend, not just hover tooltips.
