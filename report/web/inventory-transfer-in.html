Transfer In

The receiving side of the transfer system. Shows all incoming transfer slips destined for the current location. Each slip can be marked as received. Uses the same component as Transfer Out in "receive" mode — no slip creation form is shown.

Incoming Slips Table

Incoming Transfer Slips List
Data Table
Shows transfer slips created by other locations that are destined for this location. Columns: slip number, sending location, item count, created date, status. Each slip shows a "Mark as Received" button.
Showing only incoming transfers (not all transfers in the system) keeps the list focused and relevant for the receiving team.
There is no way to verify what was actually received vs what was expected. Marking a slip as "received" is all-or-nothing — if a transfer of 15 items arrives with only 13, the system has no way to record the 2-item shortage.
Add an optional scan-to-verify step before marking received: "Scan received items or confirm all X items received without scanning." If scanning reveals a short count, record the discrepancy. At minimum, add a notes field on the receive action: "Received 13 of 15 — 2 items missing."
Mark as Received Button
Action Button
Changes the slip status from "in-transit" to "received" and updates all included items to "in-stock" at this location. No confirmation shown. No partial receipt supported.
No confirmation before changing item statuses. Clicking "Received" on the wrong slip changes the status of someone else's items to in-stock at the wrong location.
Show a confirmation: "Mark Transfer #70043 (15 items from Main Warehouse) as received? This will add all items to your in-stock inventory." Include the sending location name prominently so the user can confirm they have the right slip.

Overall Assessment

No partial receipt support means the system cannot accurately record reality — transfers in real warehouses routinely arrive with shortages. Every transfer is recorded as "all items received" even if some are missing, creating phantom inventory that will show up as missing items later during cycle counts.
Key improvement: Add a notes/quantity field on the receive action to capture shortages. Even without a full scan workflow, letting a receiving staff member type "received 13 of 15" and recording that discrepancy would be a significant data quality improvement.
