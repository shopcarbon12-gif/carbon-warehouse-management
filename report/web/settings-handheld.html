Handheld Settings

Controls behavior of the CarbonWMS mobile app on handheld RFID devices. Settings here are pushed to all handhelds at the active location. The page uses toggle switches for each setting. Changes are saved with a Save button.

Toggle Switches

Individual Toggle Switches (multiple)
Toggle Switches
A list of on/off toggles controlling handheld behavior. Each toggle has a label. Settings are loaded from the server when the page opens. Changing a toggle only updates the local UI state — changes must be saved with the Save button to take effect on devices. Examples of toggle settings: auto-read scanning, beep on read, confirmation dialog before committing, show item names vs EPCs only, etc.
Suggestion: Each toggle should have a brief description below the label explaining what it does when on vs off. For example: "Auto-read: When ON, the reader scans continuously. When OFF, each scan must be manually triggered." Currently toggles show labels only, which can be ambiguous.
Save Button
Primary Button
Saves all toggle changes to the server. Shows "Saving…" while in progress. After saving, a success or error message appears. Changes take effect on connected handhelds at their next sync (typically when the app refreshes its settings).
Suggestion: The Save button should be disabled (greyed out) when there are no unsaved changes. Also, after saving, show clearly: "Settings saved — handhelds will receive these changes at their next sync." This removes uncertainty about when changes apply.
Loading / Error States
States
While loading current settings, the page shows a loading state. If loading fails, an error message appears. The settings are fetched fresh each time the page opens.
Suggestion: If the settings fail to load, the toggles might show incorrect default states. Make sure to clearly disable all toggles and show an error message if the settings could not be loaded, rather than showing potentially wrong defaults that a user might save accidentally.

Overall Handheld Settings Suggestions

1. Per-device vs tenant-wide: Currently these settings apply to all handhelds at a location. Consider allowing per-device overrides for specific use cases (e.g., one handheld used for receiving might need different settings than one used for cycle counts).

2. Settings preview: A "Preview on handheld" simulation showing how the app will behave with the selected settings would be very helpful for new admins.

3. Rollback: Allow reverting to the previous saved settings in case a change causes problems on the floor.
