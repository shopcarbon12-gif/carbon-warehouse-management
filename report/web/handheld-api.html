Handheld Batch API Documentation

A minimal developer documentation page embedded in the app itself. Describes the API endpoint that handheld devices use to upload batches of scanned EPC codes to the server. This page is for developers integrating with the system, not for warehouse staff.

Page Content

Authentication Instructions
Documentation Text
Explains that API requests must include a header called "X-WMS-Device-Key" whose value matches a secret environment variable called "WMS_DEVICE_KEY" configured on the server (in Coolify, the deployment platform). No further authentication method is described.
Suggestion: The authentication section mentions "Coolify" by name — this is a deployment-platform-specific detail that may not apply to all installations. Replace with a generic reference: "Set in your server's environment variables." Also, the docs do not explain how an admin obtains or rotates the WMS_DEVICE_KEY — add a brief note explaining where to find this value and what to do if it needs to be changed.
POST /api/handheld/batches Endpoint Docs
Documentation Text
Describes a single API endpoint. The request body must be JSON with three fields: a batch_id (a unique identifier that allows safe retry — sending the same batch_id twice won't create duplicates), a location_id (the UUID of the warehouse location where the scan happened), and epcs (an array of 1 to 5000 RFID tag codes). The response contains whether it succeeded, whether it was a duplicate (already received), how many EPCs were accepted, and the batch ID. The server stores the batch and then processes it in the background.
Suggestion: The documentation is minimal but covers the essentials. Key improvements needed:
1. Show a concrete example request body, not just a field list — developers should be able to copy-paste to test.
2. Describe error responses — what HTTP status codes are returned for authentication failure, invalid data, or server errors?
3. Explain "location_id" — how does a developer find the UUID for a location? (They need to look it up from another API or the database — this should be documented.)
4. Describe what "enqueues handheld_process sync job" means from an operational standpoint — how long does processing take? Is there a way to check if a batch has been processed?

Page Design & Placement

Page Location (separate from main app)
Page Structure
This page lives at the URL path /docs/handheld-api — notably outside the main app's authenticated layout (the sidebar navigation). It appears to use a different layout (or no layout) with a simple centered monospace text format.
Suggestion: Since this page is outside the main navigation, it may be publicly accessible without login. If the API documentation contains any sensitive information (like the name of security headers), it should be protected behind authentication. Verify whether this page requires login to view.

Overall Handheld API Docs Suggestions

1. Only one endpoint documented: The mobile app almost certainly calls more than one server API (for settings, OTA updates, status checks, etc.) but only the batch upload endpoint is documented here. The docs should cover all mobile-to-server API calls.

2. No versioning or change history: If the API changes, there is no indication of what version this documentation describes or when it was last updated. Add a simple "Last updated" date or an API version number.

3. No navigation or context: This page has no back link, no sidebar, and no indication that it is part of CarbonWMS. Add minimal navigation — at minimum a link back to the main app and a note that this is the CarbonWMS developer documentation. A developer who lands on this page via a direct link would have no idea what system they are looking at.
