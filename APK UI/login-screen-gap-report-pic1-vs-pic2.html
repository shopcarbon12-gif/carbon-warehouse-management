1. Executive summary

Pic 1 shows the shipped login screen after the last implementation pass. It is close to Pic 2 in structure (logo, title, subtitle, server row, email, password, checkboxes, teal SIGN IN), but several Material Design side effects and one layout strategy diverge from the flat mock.

  • Primary visual gap: Email and password fields show an inner white “pill” or nested field with border (Pic 1). Pic 2 shows a single light-gray rounded tray only — no inner white box, no extra outline (except optional focus ring not visible in static mock).
  • Password focus (Pic 3): A bright green border around the password inner area is visible — not present in Pic 2 (teal/gray system only). This is consistent with Flutter’s default InputDecorator / theme focus border when Material 3 or a theme color maps focus to green.
  • Keyboard shrink (Pic 3): When the IME opens, the entire form scales down (logo, text, fields, button). Root cause in code: FittedBox(fit: BoxFit.scaleDown) inside an Expanded that receives a reduced max height when the scaffold resizes for the keyboard (resizeToAvoidBottomInset: true by default).
  • Biometric (Pic 4): The exact snackbar text matches the post-login enrollment path in login_screen.dart when the user cancels or local_auth fails. The app then still proceeds to the dashboard because enrollment is optional. “Not working” may mean enrollment fails, vault empty, or users expect biometric on dashboard without prior successful enrollment.

3. Detailed visual gaps

3.1 Nested “double box” on email and password (Pic 1)

Pic 2 describes fields as light gray containers with rounded corners only — “no borders” except the outer fill shape. Pic 1 shows a secondary inner shape (white, thin dark border) that reads as a different component inside the tray.

  • Likely cause: Flutter TextField / InputDecoration default filled decoration, or theme filled: true with a white fillColor inside a parent gray Container.
  • Target: Transparent input background inside the gray tray; no visible inner border in default state; focus state should match brand (teal/neutral), not system green.

3.2 Focus / error border color (green on password, Pic 3)

The mock’s accent is teal (~#1B7D7D / #1E7E7E). A green focus ring suggests the app theme’s colorScheme.primary or M3 InputDecorator mapping is green, or the field is in an “error” or “focused” state styled by the platform theme.

3.3 Typography micro-gaps

  • Label row: “SERVER URL” / “CONNECTED VIA SSL” weight and size vs mock (minor).
  • Subtitle letter-spacing vs Pic 2 (minor).

3.4 Logo treatment

Pic 2 illustration uses a rounded tile with gradient; the approved production logo is a square black field with line art. Implementation correctly uses the asset without tinting. Any “gap” vs Pic 2 is intentional brand vs placeholder.

4. Bug: keyboard open — full screen shrinks (Pic 3)

Symptom: When the soft keyboard opens, the entire login UI (logo, typography, fields, button) appears to scale down into the reduced viewport, with tight gaps and almost no padding above the keyboard.

4.1 Technical root cause (current codebase)

  • Scaffold defaults to resizeToAvoidBottomInset: true, so when the keyboard appears, the body’s available height shrinks.
  • The login form uses Expanded → LayoutBuilder → FittedBox(fit: BoxFit.scaleDown) wrapping the main Column.
  • FittedBox.scaleDown uniformly scales its child to fit the constraints. Smaller max height ⇒ smaller scale factor for everything — hence “whole screen shrink.”

4.2 Direction for fix (for approval — not implemented)

  • Remove FittedBox for the common case, or gate it only when keyboard is closed.
  • Consider resizeToAvoidBottomInset: false on this screen and pad bottom by MediaQuery.viewInsets.bottom so content translates instead of scales.
  • Alternatively, allow a bounded scroll only while the keyboard is open (product previously required no scroll — trade-off to confirm).

5. Bug: biometric “not working” (Pic 4)

Pic 4 text: “Biometric verification was cancelled or failed. You can try again from Handheld settings.”

5.1 Where this string comes from

This message is shown from login_screen.dart in the post–successful-login biometric enrollment flow: after the user taps “Turn on” in the dialog, LocalAuthentication.authenticate returns false or throws — not from the dashboard module list itself.

5.2 Why it can appear “on the dashboard” (Pic 4)

Enrollment runs after login succeeds and before navigation to the dashboard completes. A SnackBar is shown and the app then calls onSuccess(). The snackbar overlays the next route, so it is visible on top of the modules screen. That matches Pic 4.

5.3 Likely failure modes to verify in a fix pass

  • User cancelled the system biometric sheet.
  • Device has no enrolled fingerprints / face.
  • local_auth Android configuration (e.g. FragmentActivity, manifest permissions, biometric prompt theme).
  • Session token read timing: enrollment stores token from getSessionToken() after login — if null, vault may stay empty silently in some branches (separate from this snackbar).
  • After logout, vault is cleared — user must enroll again; expectations may mismatch.

5.4 Direction for fix (for approval — not implemented)

  • Reproduce on device with logging around authenticateWithBiometric and secure storage writes.
  • Align UX: if enrollment fails, optionally show a shorter message or retry on login only (avoid confusing dashboard context).
  • Confirm Android MainActivity extends FlutterFragmentActivity if required by local_auth on this Flutter version.

6. Proposed remediation outline (pending approval)

  1. Visual parity: Remove inner white filled input decoration; single gray tray; teal/neutral focus only; match Pic 2.
  2. Keyboard: Eliminate scale-down-on-inset; prefer inset padding or controlled scroll; keep SIGN IN usable above keyboard.
  3. Biometric: Debug local_auth on target devices; tighten enrollment success criteria and messaging; verify vault token persistence.
